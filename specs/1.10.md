## Problem
Distribution today is `git clone` + force-copy installer + `git pull` to update — no versioning. Spec 1.2 chose a plugin (`/plugin install`) as the primary distribution for skills/agents/commands, with the installer retained for hooks and as fallback, retired only when criteria are met (D5).

## Value
As a flow user across several projects and two Claude profiles I want versioned `/plugin install` updates for skills/agents/commands, without losing the always-on hook seatbelt during the transition, so that updates are reliable and reversible.

## Scope
**In:** author the plugin manifest + marketplace bundling skills/agents/commands; keep `install.{sh,ps1}` for hook registration + fallback (in lockstep); document the criterion-gated transition (retire installer only once the plugin registers hooks reliably across both profiles AND all in-use projects migrated); **handle stale copied commands** — commands migrated to skills (flow/flow-hunt/flow-review/flow-pr) linger in each profile's `commands/` because the copy-installer never prunes (harmless today: skill outranks stale command; from 1.11). Plugin packaging manages its own file lifecycle and should moot this; if the installer is still primary, add a prune step.
**Out:** the skill/agent content itself (1.7–1.9, 1.11) — this packages what those produce.

## Acceptance criteria
- [ ] Plugin manifest + marketplace installs skills/agents/commands via `/plugin install`.
- [ ] Installer still registers hooks and works as fallback; `install.sh`/`install.ps1` stay in lockstep.
- [ ] Transition exit criteria documented in README; no fixed calendar date.
- [ ] Hooks (`hooks`/`permissionMode`/`mcpServers`) confirmed to remain profile-scoped as required.

## Plan (thin slices)
1. [ ] Plugin manifest + marketplace; verify `/plugin install` across both profiles.
2. [ ] Confirm hook registration path (plugin vs installer) and document the split.
3. [ ] README: install via plugin, transition criteria; keep installer docs.
