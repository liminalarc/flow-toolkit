## Problem
`flow-review` is a monolithic command that loads whole and audits four lenses (docs/UX/marketing/product) serially in one context. Spec 1.2 chose it as the first skill migration — its lenses are the cleanest independent, read-only fan-out and the most natural progressive-disclosure split.

## Value
As a flow user I want `/flow-review` to load only the lens rubric I invoke and to fan the audit out to one reviewer agent per lens with the main thread synthesizing, so that reviews are faster, context stays lean, and the skill + parallel-reviewer pattern is proven for later migrations.

## Scope
**In:** convert `flow-review` command → skill (`SKILL.md` + one `reference/<lens>.md` rubric per lens); reviewer sub-agent (read-only) dispatched per lens; main-thread synthesis; entry point stays `/flow-review`.
**Out:** `flow-pr` (1.9), `flow`/`flow-hunt` (1.11); plugin packaging (1.10).

## Acceptance criteria
- [ ] `flow-review` is a skill; each lens rubric is a `reference/*` file loaded only on that path.
- [ ] One reviewer agent per lens runs in parallel; main thread synthesizes a single report.
- [ ] `/flow-review` behaves identically at the entry point (no user-visible regression).
- [ ] README + `commands`/skill docs updated.

## Plan (thin slices)
