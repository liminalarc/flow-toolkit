## Problem
Every existing flow agent is static — `flow-verifier`/`flow-pr-reviewer` read a diff, `flow-reviewer`
reads source. **None runs the app.** A class of interface defects only surfaces when you *use* it: does
a screen match the design system, and can a user finish the intended flow without friction. This is the
**Validate** step of engineer-directed delivery pointed at the interface — the **live-driving complement**
to `review`'s static UX lens.

## Value
As an engineer directing agentic delivery I want an agent that drives the running app and scores its UI/UX
against a rubric, so interface regressions and flow friction are caught in the Validate step — under my
gate — not by me clicking around after the fact.

## Scope
**In:**
- **`flow-ux-validator`** agent (read-only re project files; `Read/Grep/Glob/Bash`) — the reusable primitive.
  Per run: applicability-check → resolve rubric → drive app (Playwright-first, vision fallback) → capture
  screens to scratch → score **ONE** lens → prioritized findings. Never edits/fixes project files.
- **`/flow:validate <screen|flow> --intent "<outcome>" [--ui|--ux] [--design-system <path|url>]`** — thin
  launcher: pick lens(es), dispatch the agent (no flag ⇒ both, serialized), synthesize one assessment. The
  human entry point into UI/UX review.
- Two baseline rubrics — `reference/ui.md` (design-system conformance) + `reference/ux.md` (Nielsen
  heuristics + WCAG + intent/friction) — and `reference/driving.md` (how to drive; loaded only on that path).
- **Applicability-aware:** non-UI repo (no runnable UI / run cmd / driver) ⇒ clean `not applicable: <reason>`
  exit — never a hallucinated critique.
- Docs: README, `docs/guide.md`, `docs/how-it-works.md` (dispatch diagram), `docs/architecture.md` (ADR:
  the live-driving agent), root `CLAUDE.md` catalog + `/flow:validate` usage examples.

**Out:**
- `/flow:run` **done-gate** integration → **1.15** (depends on 1.14 — the gate needs the agent first).
- **Persisted / self-maintaining project rubric** (`.flow/validate/*.md`, bootstrap + drift-refresh) →
  **1.16**. 1.14 takes project specifics as per-run args (`--design-system`/`--intent`, or infer from source).
- Closed-loop auto-fix (agent validates + reports; the orchestrator/engineer owns any fix loop).
- PR-review caller (`/flow:pr`) — judged overkill; deliberately not specced.
- Replacing `review`'s static UX lens; a visual-regression baseline system (reuse the target's if present).

## Acceptance criteria
- [ ] `flow-ux-validator` drives a real running app (Playwright, vision fallback) and captures screens/interactions — proven against a scratch app
- [ ] UI findings cite design-system conformance; UX findings cite intent + heuristics/WCAG — proven by a scored run naming a planted defect
- [ ] Output is a prioritized read-only assessment (never edits) — inspection
- [ ] Runs are scope-bound (one screen/flow) and rubric-driven (baseline `reference/*` + explicit project specifics) — inspection
- [ ] Non-UI repo ⇒ clean "not applicable" exit, no hallucinated critique — proven by pointing it at this repo (flow-toolkit, no app)
- [ ] Docs updated: README, guide, how-it-works, architecture, CLAUDE.md catalog + usage example — inspection

## Decisions
- 2026-07-18 — **Agent vs skill.** The agent is the reusable primitive; the skill is a thin launcher.
  Applicability-check + rubric-resolution live in the **agent** so every caller (validate skill now,
  `/flow:run` in 1.15) reuses identical behavior — skills don't call skills.
- 2026-07-18 — **Named `flow-ux-validator`, not "validator".** Scoped to UI/UX, distinct from the static
  `flow-verifier`/`flow-pr-reviewer` (which own correctness + clean code). This is the only agent that
  drives the live app.
- 2026-07-18 — **Rubric = layered.** Baseline heuristics ship in `skills/validate/reference/{ui,ux}.md`;
  project specifics (design system, intent) are explicit input. 1.14 passes them as per-run args (or infers
  from source); a **persisted, self-maintaining project rubric** (`.flow/validate/*.md`) is **1.16**, read by
  both the standalone skill and 1.15's gate.
- 2026-07-18 — **Driving.** Playwright-first (deterministic, via Bash); computer-use/vision documented
  fallback. Applicability-aware so non-UI repos exit N/A.
- 2026-07-18 — **Lifecycle gate split to 1.15** (the gate needs the agent first). PR-review caller judged
  overkill; not specced.
