## Problem
flow-toolkit's `review` skill has a **UX lens**, but it audits code / current state read-only — it never
drives the running app. A whole class of defects only shows up when you actually *use* the thing: whether
a screen matches the design system, and whether a user can complete the intended flow without friction.
This is the **validation** half of engineer-directed agentic delivery (Understand → Change → Validate →
Capture) applied to the interface — an agent that *runs the app*, not just reads it. Conceptually it's the
eval-harness pattern (baseline → score → track) pointed at UI instead of prompts.

## Value
As an engineer directing agentic delivery I want an agent that drives the running app and validates its UI
and UX against a rubric, so that interface regressions and flow friction get caught in the Validate step —
under my gate — instead of by me clicking around after the fact.

## Scope
**In (when picked up):**
- A **validation agent** (agents/ + a skill entry) that **drives the running app** (Playwright for
  deterministic flows, or computer-use/vision) — real interactions + screenshots, never a static read.
- **Two lenses, two rubrics:**
  - **UI** — does the screen conform to the **design system** (layout, components, states, spacing,
    consistency). Mostly per-screen inspection.
  - **UX** — can a user complete the **intended flow**, and how much friction (step count, navigation/IA
    clarity, error recovery, discoverability). Scored against the **engineer-defined intent/outcome** +
    **established UX best practices** (heuristics, accessibility). Requires driving the real task.
- Output = a prioritized **UX/UI assessment** (findings + severity) — the shape the orchestrator hands
  back to Change under the engineer's gate. Read-only by default (judges, never fixes).
- **Grounding + safety:** actually drive the app (no hallucinated critique); bound each run to one
  screen/flow; the rubric is explicit input (a vague rubric produces slop — same lesson as LLM-judge rubrics).

**Out:**
- **Closed-loop auto-fix** (edit code → redeploy → re-observe). The orchestrator/engineer owns the loop;
  this agent validates and reports. Auto-fix is a separate, later spec.
- Replacing the static `review` UX lens — this is the live-driving complement, not a rewrite.
- A visual-regression baselining system of its own — reuse the target project's if present.

## Acceptance criteria
- [ ] The agent drives a real running app (Playwright/computer-use) and captures screens/interactions — proven against a scratch app
- [ ] UI findings cite design-system conformance; UX findings cite intent + best-practice heuristics — proven by a scored run
- [ ] Output is a prioritized read-only assessment (never edits) that fits the Validate → Change hand-back — inspection
- [ ] Runs are scope-bounded (one screen/flow) and rubric-driven (rubric is an explicit input) — inspection

## Decisions
- 2026-07-18 — Captured from the "engineer-directed agentic delivery" discussion (Miro-board Validate step).
  UI vs UX split is deliberate: UI = design-system conformance (inspect a screen); UX = task completion +
  friction (drive the flow), scored against engineer intent + established UX best practices. Iteration is
  the **orchestrator's** loop, not the agent's — the agent validates and reports, like a read-only reviewer.
  Closed-loop auto-fix explicitly deferred. Parallel to LitmusAI's prompt-eval thesis (score an artifact
  against a rubric, track movement).
