## Problem
`/flow-ship`'s spec-DONE gate derives the release's spec set from `[#id]` commit-*subject* tags, but nothing ensures they're present — 1.1/1.3 shipped with untagged subjects, forcing the weaker index fallback. Make the tag reliable.

## Value
As a maintainer cutting releases, I want commits to reliably carry their spec id so that `/flow-ship` derives the release's specs mechanically from commits, not by guessing from the index.

## Scope
**In:**
- `/flow` — when a spec is in play, emit `[#<id>] type: subject` (the leading bracket the commit guard already accepts); id in the subject, not the body.
- `flow-commit-guard.sh` — local mode, **exactly one** IN PROGRESS spec, staged source, message lacks any `[#…]` tag → **nudge** (non-blocking) with the exact `[#<id>]` to prepend. Extends the existing spec-less nudge; silent when a tag is present, when 0 or >1 specs are IN PROGRESS, or no source is staged.
- `/flow-ship` — derive `[#id]` from commit subject **or** body; index fallback only when none exists anywhere.

**Out:** rewriting the commit message (the hook nudges, never edits the command); ado work-item tagging (already documented); blocking (chose nudge).

## Acceptance criteria
- [x] `/flow` docs instruct `[#id] type:` **subject** tagging.
- [x] Guard nudges the exact `[#<id>]` when one spec is IN PROGRESS + no tag present; silent when tagged, 0/>1 IN PROGRESS; never blocks — `hooks.test.sh` covers it.
- [x] `/flow-ship` finds the id in subject or body before any index fallback.
- [x] README documents the auto-tag behavior.

## Decisions
- 2026-07-14 — **Nudge, not block** — handles merge/chore/multi-spec commits gracefully; matches the toolkit's documented "no cry-wolf" hook philosophy (same reason the Stop-nudge was rejected).
- 2026-07-14 — Nudge only when **exactly one** spec is IN PROGRESS (local) — >1 is ambiguous, 0 already covered by the spec-less nudge.

## Verification / evidence
- **Guard nudge, direct smoke** — untagged `docs:` subject with spec 1.4 IN PROGRESS emitted the `additionalContext` note naming `[#1.4]` (exit 0); a `[#1.4]`-tagged subject was silent (exit 0).
- **`hooks.test.sh` → 41 passed** (was 34): 4 new commit-guard cases — untagged nudges the exact id, tagged is silent, >1 IN PROGRESS is silent (ambiguous), non-conventional still blocks (exit 2). First commit-guard coverage in the harness.
- **Live dogfood** — every commit of this spec (`753c64a`, `ae3c4bc`, `ccd508b`) is subject-tagged `[#1.4]`, so a future `/flow-ship` derives this spec from `git log` mechanically.
- **`/flow-ship` derivation** — prompt change (no code): now scans subject then body (`git log --format=%B`) before any index fallback.
