## Problem
Spec detail files bloat in real use: sections restate each other (Value ≈ Problem, Scope ≈ AC, Plan ≈ AC), prose runs long, and the Progress log grows unbounded. Every read pays that cost in context — the exact waste the toolkit exists to prevent.

## Value
As a developer using flow, I want spec files to stay terse and non-repetitive so that reading a spec costs minimal context and its key detail stays easy to find.

## Scope
**In:** brevity + no-restatement guidance in the `--add`/init templates; a terse append-only Progress-log convention; a soft (non-blocking) bloat warning in `flow-spec-guard.sh` and `/flow-lint`, with a configurable budget.
**Out:** task breakdown (→ 1.2); changing which sections a spec has.

## Acceptance criteria
- [x] `flow.md` (`--add`) and `flow-init.md` templates state a per-section brevity target and a "each section says what no other does" rule — no restatement.
- [x] `flow-spec-guard.sh` emits a WARN (never a block) when `specs/<id>.md` exceeds a line budget from `.flow-toolkit.json` (alongside the CLAUDE.md caps); default applies if unset.
- [x] `/flow-lint --specs` reports the same warning; `hooks.test.sh` covers it.
- [x] README documents the terseness rules and the new budget knob.

## Decisions
- 2026-07-14 — Bloat check is a **soft WARN**, never a block — specs legitimately vary; blocking would fight real specs.
- 2026-07-14 — Budget default **120 lines**; config key `spec.maxLines` in `.flow-toolkit.json`, mirroring the `claudeMd` caps shape.
- 2026-07-14 — Warning emits as a PostToolUse `additionalContext` note in hook mode, stderr in direct mode, so it never corrupts `flow-commit-guard.sh`'s stdin contract.

## Verification / evidence
- `bash hooks/hooks.test.sh` → 34 passed, 0 failed (over-budget warns+passes, under-budget silent, hook-mode JSON note, `spec.maxLines` override).
- Real-repo smoke: a 155-line spec warned in both modes and exited 0; a 500-line `spec.maxLines` silenced it; spec 1.1 (under budget) stayed silent.
