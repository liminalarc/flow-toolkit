## Problem
The toolkit is 7 monolithic command files + 4 hooks. It can't do specialized work in isolated context, big specs have nowhere to put task detail, and command files load whole (bloat). Claude Code now offers **skills** (progressive disclosure), **sub-agents** (isolated context), and **plugins** (versioned distribution) — the toolkit uses none of them. Goal: a more autonomous, self-contained workflow that still keeps the human plan-approval checkpoint.

## Value
As a flow user I want the toolkit to break specs into tasks, dispatch specialized work to sub-agents, and load only the detail each path needs, so that large specs are tractable, sessions stay lean, and routine work can run with less hand-holding — without losing control of direction.

## Scope
**In (this spec = the *design*):** decide the target architecture and spawn the implementation backlog. Covers: (a) spec→task file model, (b) command→skill migration where intentional, (c) the sub-agent catalog incl. verifier agents, (d) layered autonomy controls, (e) plugin packaging.
**Out:** building it — each decided piece becomes its own NOT STARTED implementation spec.

## Target architecture (to be ratified by this spec)

**a. Spec → tasks (the #2 model).** A spec earns a directory only when big enough; small specs stay flat.
```
specs/1.7/
  1.7.md      # orchestrator: Problem/Value/Scope/AC + task list
  1.7.T1.md   # task: the "how" only
  1.7.T2.md
```
The spec holds why/what; tasks hold how. Touches every `specs/<id>.md` assumption: spec-guard globbing, `/flow-lint` orphan/archival logic, `flow-preflight.sh` `to`-resolution, archival (move the whole dir).

**b. Commands → skills, by intentionality.** Convert where progressive disclosure pays: `flow` (the `--add` template, `--ideas` lenses, `--clean` rules become `references/*` loaded per path), `flow-hunt`, `flow-review`, `flow-pr`. Thin/deterministic ones (`flow-lint`, `flow-ship`) can stay commands. Commands and skills both produce `/name`, so the entry points are unchanged.

**c. Sub-agent catalog (isolated context, read-only unless building).**
- **Parallel reviewers** — one agent per `flow-review` lens / `flow-pr` dimension; main thread synthesizes.
- **Research fan-out** — `flow-hunt` personas × dimensions.
- **Per-task implementers** — one agent per task (from **a**), against an approved plan/contract.
- **Verifier agents** — independent agent checks an implementer's diff against the task AC *before integration* (the "agents reviewing agents" ask; the adversarial-verify pattern).
- **Spec-writer** — drafts specs and enforces 1.1 terseness.

**d. Layered autonomy controls.** Precedence: project override > repo default > per-spec.
- Per-spec front-matter: `autonomy: checkpoint | auto-build` (default `checkpoint`).
- Repo default in `.flow-toolkit.json`: `autonomy.default`.
- Project override: `.flow-toolkit.json` `autonomy.force: auto-build` overrides all front-matter.
- **What autonomy controls = whether flow pauses for *plan approval*.** It cannot bypass Claude Code's permission system, edit config, or self-approve (docs-confirmed). Verifier agents (**c**) are the safety net that makes `auto-build` tolerable.

**e. Distribution: plugin.** Bundle skills + agents + hooks + a marketplace; `/plugin install` replaces the force-copy installer with versioned updates. Gotcha: plugin sub-agents can't declare `hooks`/`permissionMode`/`mcpServers` — those stay in profile `.claude/`. Likely runs in parallel with the install script during transition.

## Decisions
- **One combined architecture spec** (user choice) rather than separate #2/#3 specs.
- **Task files live in `specs/<id>/`** (user choice).
- Autonomy is spec-specific + repo default + project force override, with verifier agents (user direction).
- This spec stays flat `specs/1.2.md` for now; it becomes the first to migrate to the `specs/<id>/` model once **a** ships (dogfood).

### Ratification session (2026-07-14)
- **D1 — Breakout threshold = manual + documented guideline; no enforcement.** Hooks support both flat `specs/<id>.md` and dir `specs/<id>/` shapes, but a hook enforces neither. Command prose advises breakout at "≥3 tasks, or a task carrying its own AC." Rationale: "thin slices, no premature abstraction" — zero multi-task specs exist today; revisit enforcement only if drift appears.
  - **Added scope:** flow is already used across ~4-5 projects, so we need a **migration path** for existing flat specs → the directory model. To be spawned as its own implementation spec, likely extending `/flow-lint --migrate`. (Not a deferral — an addition realized as a spawned spec.)
- **D2 — Task files carry a *local* AC; deferrals + DONE-gating stay on the orchestrator only.** A task file = the "how" plus a local "done when X" contract (the seam a per-task implementer builds to and a verifier agent checks against). `flow-preflight.sh` (`resolved`/`wellformed`) and the commit guard keep pointing at exactly one file per spec (the orchestrator `<id>.md`), never rolling up task files. Reconciles piece (a) "tasks = how" with piece (c) "verifier checks task AC" while preserving single-source gating.
- **D3 — Verifier gating is mode-dependent, single verifier.** Blocking under `auto-build`, advisory under `checkpoint` — the verifier is the safety net exactly where no human watches. One verifier per task; a failing `auto-build` verdict → one bounded retry → escalate to human (that task falls back to checkpoint). Majority-vote / N-skeptic panels are a documented future refinement, not v1.
- **D4 — First skill migration = `flow-review`, then `flow-pr`.** Its four independent, read-only lenses are the cleanest fan-out and the most natural per-lens `reference/*` split — lowest-risk proof of skill + parallel-reviewer + synthesis.
- **D5 — Distribution = both, criterion-gated transition.** Plugin primary for skills/agents/commands; installer stays for hook registration + fallback. Retire the installer only once the plugin registers the toolkit's hooks reliably across both profiles AND every in-use project has migrated — not a fixed date.

### Command vs. skill criterion (recorded for the migrations)
Skill when the prompt is large + path-dependent (progressive disclosure pays) or should be model-reachable by context — `flow`, `flow-hunt`, `flow-review`, `flow-pr`. Command when thin and deliberate-only, never auto-triggered — `flow-lint`, and especially `flow-ship` (release must never fire by context). Entry points (`/name`) are identical either way.
