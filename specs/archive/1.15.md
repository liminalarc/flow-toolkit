## Problem
1.14 builds `flow-ux-validator` + `/flow:validate` as a standalone entry point. But live UI/UX validation
is most valuable when it isn't optional — a spec touching the interface shouldn't reach `DONE` with
unresolved UI/UX findings. This wires the **same agent** into the build lifecycle so validation happens
under the engineer's gate, automatically, instead of only when someone remembers to run `/flow:validate`.

## Value
As an engineer directing agentic delivery I want a spec to declare it needs UI/UX validation and have
`/flow:run` exercise `flow-ux-validator` at the done-gate, so interface findings are surfaced and triaged
before `DONE` — not skipped when I forget to run `/flow:validate`.

## Scope
**In:**
- A `validate:` front-matter block on a spec declaring the target/intent (+ optional lens, design-system).
- `/flow:run`'s done-checklist dispatches `flow-ux-validator` (the 1.14 agent) for the declared target,
  surfaces findings, and requires them triaged/resolved before flipping `DONE`.
- Sources project specifics without one-off args: the **spec declares** the target/intent; the
  design-system rubric comes from **1.16**'s persisted `.flow/validate/*.md` when present.

**Out:**
- The agent itself + the standalone skill (1.14).
- Closed-loop auto-fix (still out).
- Making validation mandatory for every spec — it's **declared per-spec, opt-in**.

## Decisions
- 2026-07-18 — **Split from 1.14.** The gate needs the agent to exist first (thin-slice sequencing). Reuses
  the 1.14 `flow-ux-validator` **directly** — `/flow:run` dispatches the agent, not the `/flow:validate`
  skill (skills don't call skills).
- 2026-07-18 — **Gate is prompt-level, not a new hook.** It lives in `implement.md` §4 beside the smoke-test
  step. `flow-spec-guard.sh` doesn't allowlist front-matter keys, so `validate:` needs no guard change and no
  `hooks.test.sh` case — thin slice, no premature machine-check.
- 2026-07-18 — **Triage reuses the deferral protocol.** A finding is fixed here or explicitly re-homed via the
  existing `deferrals:` machinery (which already mechanically gates `DONE`) — no parallel gate.
- 2026-07-18 — **Design-system precedence:** spec's `validate.design_system` > 1.16's `.flow/validate/*.md`
  (when present) > infer from source.
- 2026-07-18 — **`NOT APPLICABLE` passes cleanly** (verdict shown), never blocks — declaring `validate:` on an
  undrivable repo is a mistake to surface, not a hard stop.
