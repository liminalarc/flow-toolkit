## Problem
1.1 makes *new* specs terse and *warns* when a spec drifts over budget, but existing specs across live projects predate those rules and can only be fixed by hand. And the soft budget is a line count — it can't see cross-section restatement or prose that should be bullets. A condense pass (one spec, a whole `specs/` dir, or a report-only audit) closes both gaps.

## Value
As a developer running flow across several existing projects, I want a command that rewrites bloated specs to the terseness rules — or just reports where they fall short — so that I can migrate a backlog to the standard and keep it there, without hand-editing or losing detail.

## Scope
**In:** `/flow --condense [<id> | --all] [--check]`.
- `<id>` condenses one spec; `--all` sweeps every `specs/<id>.md` (project-wide migration).
- Rewrite applies 1.1's rules losslessly (one job per section, no cross-section restatement, prose→bullets); **copies the Progress log verbatim** and preserves every AC's substance; shows a diff and confirms **per spec** before any write; echoes a no-op when already terse; amends **freely regardless of index status**.
- `--check` is report-only: audits against the *full* terseness rules (budget **and** the qualitative restatement/repetition rules a line count can't catch) and reports findings without proposing a rewrite — usable on already-condensed specs as ongoing hygiene.

**Out:** format migration (`/flow-lint --migrate` owns legacy inline → index model); the terseness rules + soft budget warning themselves (1.1); interactive field-level editing (`/flow <id>` resume covers ad-hoc edits).

## Acceptance criteria
- [x] Invocable as `/flow --condense <id>` and `/flow --condense --all`; echoes a no-op when a spec is already under budget + non-repetitive.
- [x] `--check` reports over-budget **and** restatement/repetition findings without writing; returns a clean report on an already-terse spec.
- [x] Diff shown and confirmed **per spec** before any write — never rewrites silently, even under `--all`.
- [x] Lossless: Progress-log entries reproduced verbatim; every AC's substance preserved (verified on a real bloated spec).
- [x] No status special-casing — a DONE/archived spec condenses the same as any other.
- [x] README + `/flow-lint --specs` cross-reference the command; README additions stay lean.

## Decisions
- 2026-07-14 — Broadened from the stub's single-spec condense to include **project-wide `--all`** and a **report-only `--check`** mode, per real multi-project use.
- 2026-07-14 — Invocation `/flow --condense`, **not** `/flow --clean` or `/flow-lint --fix`: those are *mechanical*; terseness rewrite/audit is a *judgment* pass. - 2026-07-14 — Runs **inline**, not via a spec-writer sub-agent — keeps 1.3 independent of 1.2.

## Verification / evidence
Smoke-tested against a hand-built bloated scratch spec (121 lines, deliberate cross-section restatement + prose):
- **Budget path** — `flow-spec-guard.sh` warned at 121 lines (soft budget 120, exit 0 non-blocking); after condensing it went silent (46 lines, well-formed).
- **Lossless** — Progress log byte-for-byte identical (4 entries, `diff` clean); all 4 ACs preserved with every substance keyword intact (top-right, PDF, CSV, charts, title, date range, timestamp, header row, one row per data point, row count, filters). 121 → 46 lines (−62%).
- **`--check` value** — the same spec at 97 lines (under budget) still carried restatement a line count can't catch, showing why the qualitative audit exists alongside `/flow-lint --specs`.
- **Guard-as-verification** — the installed spec guard blocked a mismatched-stem output file, confirming the "guard re-validates on save" claim.
- `bash hooks/hooks.test.sh` → 34 passed (no hook logic changed; command-only spec, per commands/CLAUDE.md testing = running it).
