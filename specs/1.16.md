## Problem
1.14's `flow-ux-validator` takes project specifics (the design system) as per-run args or infers them each
time — wasteful, and it never produces a rubric that *improves*. A project's design system is
stable-but-drifting; it deserves a **persisted, engineer-owned rubric** the agent reads instead of
re-deriving. This also gives 1.15's done-gate a repeatable source for the design-system layer without
re-passing args every build.

## Value
As an engineer directing agentic delivery I want the project's UI/UX rubric captured once in the repo and kept
current, so `flow-ux-validator` scores against a stable, improving standard instead of re-inferring it — and
so the done-gate has something durable to check against.

## Scope
**In:**
- A persisted project rubric in the target repo — `.flow/validate/ui.md` + `.flow/validate/ux.md` (the
  **project layer only**; baseline heuristics stay in the toolkit).
- **Bootstrap on first run** — absent rubric ⇒ the agent infers a draft from source and returns it as a
  *proposed* draft; the main thread **saves only on engineer approval**. Never silently authoritative.
- **Drift-refresh** — later runs cheaply detect the design system moved (fingerprinted `basis:` files) and
  **offer** an update. Never overwrite the curated rubric silently.
- `flow-ux-validator` reads `.flow/validate/*.md` when present (the project layer of its rubric); the
  `/flow:validate` skill + 1.15's gate consume it.

**Out:**
- The agent + standalone skill (1.14); the done-gate (1.15).
- Baseline heuristics (stay in `skills/validate/reference/`).
- Auto-editing app code to conform (validation only, never fixes).

## Acceptance criteria
- [ ] Absent rubric ⇒ agent proposes a source-inferred draft; saved only on engineer approval — proven on a scratch app
- [ ] Present rubric ⇒ agent reads it as the project layer (no re-inference) — proven
- [ ] Drift detected ⇒ agent surfaces the change and offers a refresh; no silent overwrite — proven
- [ ] Docs updated (guide, README) — inspection

## Decisions
- 2026-07-18 — Split from 1.14: rubric *lifecycle* (bootstrap / persist / refresh) is a distinct capability
  from the agent that *uses* the rubric. Persisted at `.flow/validate/*.md` in the target repo —
  engineer-owned, agent-proposed. Serves both the standalone skill (1.14) and the done-gate (1.15).
- 2026-07-18 — **Read-only seam:** the agent proposes (reads present rubric / returns inferred draft / flags
  drift); the **main thread persists** every write, confirm-first. Preserves 1.14's read-only-re-project boundary.
- 2026-07-18 — **Drift = fingerprinted basis** (`sha256`, first 12) recorded in rubric front-matter; re-hash to
  detect. Cheap + deterministic vs re-inferring each run.
- 2026-07-18 — **D3=A:** drift logic centralized in `flow-preflight.sh` (`rubric-drift` + `rubric-basis`),
  TDD-covered — the "a rule lives in one place" pattern; three consumers (agent, `/flow:validate`, gate) share it.
- 2026-07-18 — **Lifecycle division:** interactive bootstrap/refresh authoring lives in `/flow:validate`; 1.15's
  gate stays lean — reads + drift-checks, then nudges to `/flow:validate` rather than duplicating persist logic.
- 2026-07-18 — `.flow/validate/*.md` is **committed** (engineer-owned standard, like `.flow/config.yml`).
