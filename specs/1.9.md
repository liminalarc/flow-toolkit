## Problem
`flow-pr` reviews a PR/branch diff across dimensions (spec/quality/tests) in one monolithic command. With the skill + parallel-reviewer pattern proven on `flow-review` (1.8), `flow-pr` is the second migration — validating the pattern on git-diff-bound, higher-frequency work.

## Value
As a flow user I want `/flow-pr` to load only the dimension I invoke and fan the review out to one agent per dimension against the diff, so that PR reviews are faster and the pattern is confirmed on diff-coupled work.

## Scope
**In:** convert `flow-pr` command → skill (`SKILL.md` + `reference/<dimension>.md`); one reviewer agent per dimension over the branch/PR diff; main-thread synthesis; entry point stays `/flow-pr`; reuse the 1.8 reviewer *contract* via a dedicated diff-bound `flow-pr-reviewer`.
**Out:** `flow`/`flow-hunt` (1.11); plugin packaging (1.10). Skills distribute via the generic 1.8 installer globs (no new-skill wiring needed); a pre-existing single-profile `install.sh` abort the dry-run exposed was fixed here as a drive-by by user decision (see Decisions).

## Decisions
- 2026-07-14 — **Dedicated `flow-pr-reviewer`, not a shared/generalized `flow-reviewer`** (user asked for best practice). flow-pr reviewers need Bash (read `git diff`; the tests dimension runs the suite); the 1.8 `flow-reviewer` is deliberately Bash-free. Merging them forces the union of tools onto both, weakening flow-review's read-only-no-shell guarantee for no gain — premature abstraction (CLAUDE.md), and the divergent tool needs are the signal to keep them separate. `flow-verifier` is the repo's precedent for a diff-bound read-only agent with Bash. "Reuse where possible" is honored by reusing the reviewer *contract shape* (rubric-driven, read-only, prioritized findings, parallel fan-out, main-thread synthesis).
- 2026-07-14 — **Test suite runs inside the `tests` reviewer**, not the main thread — keeps "one agent per dimension against the diff" clean and avoids passing diffs inline.
- 2026-07-14 — **No new-skill installer wiring.** 1.8 made both installers glob `skills/*/`, so a new skill dir is distributed automatically; verified by dry install.
- 2026-07-14 — **Drive-by fix of a pre-existing single-profile `install.sh` abort** (user decision, given severity). The single-profile dry-run exposed that `add_profile` returned the last test's non-zero on a skip path (e.g. an unmatched `~/.claude-*` glob); under `set -e` that aborted the whole installer before installing anything — breaking any user with only `~/.claude`. Masked on this 2-profile machine. Fixed by making `add_profile` always `return 0`. `install.ps1` uses `Get-ChildItem -Filter` + `Where-Object` with no `set -e` analogue, so it never had the bug — lockstep parity confirmed by reading, no PS1 edit. Verified by a fresh single-profile dry install (EXIT 0, all artifacts land).
