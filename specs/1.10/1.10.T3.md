## Goal
With the plugin now owning commands/skills/agents/hooks, demote `install.{sh,ps1}` to a documented legacy fallback and clean up what the old copy-install left behind — resolving the [[1.11]] stale-command deferral.

- Add a **prune step** the migration path runs: remove from each profile the toolkit files the plugin now owns and the stale renamed-away files — old bare commands (`flow.md`, `flow-hunt.md`, `flow-review.md`, `flow-pr.md`, `flow-init.md`, `flow-lint.md`, `flow-ship.md`) and old skill dirs (`skills/flow`, `skills/flow-hunt`, `skills/flow-review`, `skills/flow-pr`) so users aren't left with duplicate/stale bare entry points alongside the plugin's namespaced ones.
- Keep the installer functional as a fallback for no-plugin environments, but stop positioning it as primary. Both installers stay in **lockstep** (any change lands in `install.sh` AND `install.ps1`).
- Decide + document the migration sequence for the existing projects/profiles: install plugin → verify → prune old files → restart.

## Done when
- [ ] A prune routine (in the installers or a documented one-liner per OS) removes the stale bare command files + old skill dirs from a profile, and is exercised on a scratch/dry run showing before/after.
- [ ] `install.sh` and `install.ps1` remain behavior-identical to each other (lockstep) after the change.
- [ ] The migration sequence is written down (feeds T4's migration guide) and leaves a profile with only the plugin's namespaced entry points, no stale bare duplicates.

## Notes
- Coexistence is graceful (bare + namespaced don't collide), so pruning is about hygiene/clarity, not fixing a break.
- Don't delete the installers — D5 keeps them as the criterion-gated fallback.
