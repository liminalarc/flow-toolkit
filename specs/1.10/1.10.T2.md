## Goal
Let the plugin register the four always-on hooks itself, so the copy-installer is no longer required for the seatbelt — the D5 exit criterion.

- Give the plugin a `hooks/hooks.json` that references scripts via `"${CLAUDE_PLUGIN_ROOT}"/hooks/<script>.sh` (double-quoted, per docs), covering the same events/matchers as today: PostToolUse(Edit|Write)→spec-guard + claude-guard; PreToolUse(Bash)→commit-guard; SessionStart→session-brief.
- Resolve the **dual-consumer** problem: the installer's registration uses the `__HOOKS_DIR__` placeholder, the plugin needs `${CLAUDE_PLUGIN_ROOT}`. Decide the file layout (e.g. the plugin's `hooks/hooks.json` is canonical with `${CLAUDE_PLUGIN_ROOT}`; the installer fallback keeps its own placeholder registration) without breaking either path.
- Keep the bash scripts unchanged; only their invocation path/registration differs.

## Done when
- [ ] Under `claude --plugin-dir .`, all four hooks fire: editing a malformed spec triggers `flow-spec-guard`; a CLAUDE.md over the cap triggers `flow-claude-guard`; a bad `git commit` message is blocked by `flow-commit-guard`; session start shows the `flow-session-brief` line.
- [ ] Confirmed to work when the plugin is enabled in **each** profile (both `~/.claude` and `~/.claude-company`), i.e. per-profile scoping holds.
- [ ] The installer's own hook registration still works unchanged (fallback path intact); `hooks.test.sh` green.

## Notes
- If plugin-provided command hooks can't block a tool call the way settings.json hooks do, that's a blocker to surface — the commit/spec guards must retain blocking behavior.
