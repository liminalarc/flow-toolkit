## Problem
Distribution today is `git clone` + force-copy installer + `git pull` — no versioning, and updates need a manual reinstall + restart across every profile. Spec 1.2 (D5) chose a Claude Code plugin as primary distribution, installer retained for hooks + fallback, retired criterion-gated. Research during this spec found plugins **always namespace** multi-skill invocations (`/plugin:skill`) — so a plugin can't preserve the bare `/flow` names. Decision: embrace the namespace under a short plugin name.

## Value
As a flow user across several projects and two profiles I want the toolkit as a **versioned Claude Code plugin** with a clean namespaced command set (`/flow:run`, `/flow:hunt`, …), so updates are reliable/reversible (no more `git pull` + reinstall + restart) — with a clear migration path off the old bare commands.

## Scope
**In:**
- `.claude-plugin/plugin.json` (`name: flow`, versioned) + `.claude-plugin/marketplace.json` (this GitHub repo is the marketplace).
- **Rename to the new scheme** — skills `flow`→`run`, `flow-hunt`→`hunt`, `flow-review`→`review`, `flow-pr`→`pr`; commands `flow-init`→`init`, `flow-lint`→`lint`, `flow-ship`→`ship`; update every `name:` front-matter + all internal cross-references. Agents keep `flow-*` names (internal dispatch, not user-facing).
- Plugin carries hooks via `"${CLAUDE_PLUGIN_ROOT}"/hooks/*.sh`; verify all four fire under the plugin on both profiles (**D5 exit criterion**).
- Retire the installer to a documented **legacy fallback**; add stale-file prune (**resolves the [[1.11]] deferral**); `install.{sh,ps1}` stay in lockstep.
- Docs: README (plugin install, new commands, migration, exit criteria) + CLAUDE.md (distribution architecture, `.claude-plugin/` structure) + a migration guide for existing projects.

**Out:** skill/agent *behavior* (unchanged — this repackages + renames, no new features).
