## Problem
`/flow-ship`'s CI gate reports `N/A — no CI`; hook regressions are caught only if someone remembers to run `hooks.test.sh` locally.

## Value
As a maintainer, I want `hooks.test.sh` to run automatically on every push/PR so that the `/flow-ship` CI gate is real and hook regressions are caught before a release.

## Scope
**In:** a minimal `.github/workflows/test.yml` running `bash hooks/hooks.test.sh` on push + PR to `main` — single ubuntu job, native bash, no extra deps.

**Out:** testing the `.md` commands (they're prompts, no unit tests); release/publish automation (a release stays a manual tag per CLAUDE.md); OS matrix (hooks are POSIX; Windows Git Bash parity stays covered by the portability rule, not CI).

## Acceptance criteria
- [ ] `.github/workflows/test.yml` runs `bash hooks/hooks.test.sh` on push + PR to `main`; green on the current tree.
- [ ] Workflow is minimal — one job, ubuntu, no extra dependencies.
- [ ] README notes CI runs the hook tests; CLAUDE.md/Project Structure reflect that CI now exists.
- [ ] Verified end-to-end — push the branch, watch the run via `gh`, confirm it passes.

## Plan (thin slices)
1. [ ] Write `.github/workflows/test.yml` -> <sha>
2. [ ] Push + `gh run watch` to prove it green -> <sha>

## Decisions
- 2026-07-14 — Single ubuntu runner, not an OS matrix — hooks are POSIX and the portability rule (not CI) owns Git Bash parity; a matrix is premature.
