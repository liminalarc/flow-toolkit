## Problem
Spec detail files bloat in real use: sections restate each other (Value ≈ Problem, Scope ≈ AC, Plan ≈ AC), prose runs long, and the Progress log grows unbounded. Every read pays that cost in context — the exact waste the toolkit exists to prevent.

## Value
As a developer using flow, I want spec files to stay terse and non-repetitive so that reading a spec costs minimal context and its key detail stays easy to find.

## Scope
**In:** brevity + no-restatement guidance in the `--add`/init templates; a terse append-only Progress-log convention; a soft (non-blocking) bloat warning in `flow-spec-guard.sh` and `/flow-lint`, with a configurable budget.
**Out:** task breakdown (→ 1.2); changing which sections a spec has.

## Acceptance criteria
- [ ] `flow.md` (`--add`) and `flow-init.md` templates state a per-section brevity target and a "each section says what no other does" rule — no restatement.
- [ ] `flow-spec-guard.sh` emits a WARN (never a block) when `specs/<id>.md` exceeds a line budget from `.flow-toolkit.json` (alongside the CLAUDE.md caps); default applies if unset.
- [ ] `/flow-lint --specs` reports the same warning; `hooks.test.sh` covers it.
- [ ] README documents the terseness rules and the new budget knob.

## Plan (thin slices)
1. [ ] Tighten the `--add` template in `flow.md` + the template in `flow-init.md` (brevity + no-restatement + terse Progress log).
2. [ ] Add the spec-line budget to `.flow-toolkit.json`; soft-warn in `flow-spec-guard.sh`; add a `hooks.test.sh` case.
3. [ ] Surface in `/flow-lint --specs`; update README.

## Decisions
- Bloat check is a **soft WARN**, never a block — specs legitimately vary; blocking would fight real specs.
- Default budget: start ~120 lines, tune from real specs (open).
