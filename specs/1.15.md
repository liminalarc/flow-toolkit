## Problem
1.14 builds `flow-ux-validator` + `/flow:validate` as a standalone entry point. But live UI/UX validation
is most valuable when it isn't optional — a spec touching the interface shouldn't reach `DONE` with
unresolved UI/UX findings. This wires the **same agent** into the build lifecycle so validation happens
under the engineer's gate, automatically, instead of only when someone remembers to run `/flow:validate`.

## Value
As an engineer directing agentic delivery I want a spec to declare it needs UI/UX validation and have
`/flow:run` exercise `flow-ux-validator` at the done-gate, so interface findings are surfaced and triaged
before `DONE` — not skipped when I forget to run `/flow:validate`.

## Scope
**In:**
- A way for a spec to **declare a validation target** (front-matter `validate:` block or similar —
  screen/flow + intent + design-system pointer).
- `/flow:run`'s done-checklist dispatches `flow-ux-validator` (the 1.14 agent) for the declared target,
  surfaces findings, and requires them triaged/resolved before flipping `DONE`.
- Sources the project specifics from the **spec declaration** (not one-off args) so the gate is repeatable.

**Out:**
- The agent itself + the standalone skill (1.14).
- Closed-loop auto-fix (still out).
- Making validation mandatory for every spec — it's **declared per-spec, opt-in**.

## Acceptance criteria
- [ ] A spec can declare a validation target; `/flow:run` reads it — inspection
- [ ] `/flow:run`'s done step dispatches `flow-ux-validator` and blocks `DONE` on open findings until triaged — proven on a scratch spec
- [ ] Non-UI / no-declaration spec ⇒ gate is a no-op (no friction in unrelated specs) — proven
- [ ] Docs updated (guide, how-it-works, README) — inspection
